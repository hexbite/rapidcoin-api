    Introduction

    #  _                  _
# |_)  _  ._  .  _|  /   _  . ._
# | \ |_| |_| | |_|  \_ |_| | | |
#         |

    The general flow of invoicing will be that the client site will request a new invoice from Rapidcoin from the Rapidcoin invoice route. The request for a new invoice will contain a notification_url and confirmation_url callback. Rapidcoin will return a HTTP 201 with the new invoice in JSON which will contain a deposit address for the customer. The client site will provide this address to the customer. The invoice can be an open invoice with no fixed amount set or it can be created with a fixed invoice amount. Generally speaking our clients use an open invoice and allow customers to deposit as much as they'd like into the address.

    Once the customer sends the BTC to the designated address, Rapidcoin will do a POST callback to the notification_url with the POST body as the invoice JSON (invoice will be updated with status paid and number of confirmations). Once the transaction has reached the number of confirmations_required on the blockchain, Rapidcoin will do a POST callback to the confirmation_url with the POST body as the invoice JSON (invoice will be updated with status confirmed and number of confirmations). Notification and confirmation URLs need to be acknowledged with a HTTP status code of 200, if a notification is not acknowledged it will retry every minute. The POST body for a notification and confirmation URL will be the JSON of the invoice response (see response of GET /invoices/:invoice_id).

    If a customer decides to deposit again into the same BTC address, a SUB_INVOICE will be automatically created in Rapidcoin with the same details as the original invoice but with a new invoice ID. The notification_url will be called back immediately with details of the new SUB_INVOICE (status will be paid). The SUB_INVOICE will contain the same details as the parent_invoice with the description SUB_INVOICE as well as the parent_invoice_id of the original invoice. The confirmation_url will be called back again once the transaction has reached the amount of confirmations dedicated by the confirmations_required. This will allow customers to deposit multiple times to the same BTC address if necessary.

    We currently guarantee about a week duration of watched deposits for any given BTC address in the system. General practice is that after this week the client site should force the customer to generate a new invoice and BTC address.

    Finally, once the transaction has reached 6 confirmations then the invoice will be changed to status complete and booked into the ledger for accounting purposes. No transaction has ever been reversed on the blockchain with 6 or more confirmations.

    Authentication

    To authorize, use this code:

    # With shell, you can just pass the correct header with each request
curl "api_endpoint_here"
  -H "Content-Type: application/json"
  -H "x-api-key: <API-KEY>"
  -H "x-api-version: 1.0"

    const RapidCoinApi = require("rapidcoin-api");
const config = {
  url: "https://www.rapidcoin.com/api",
  apiKey: "my-api-key",
  apiVersion: "1.0",
};

let rc = RapidCoinApi(config);

    Rapidcoin expects for the API key to be included in all API requests to the server in a header that looks like the following:

    Content-Type: application/json

    x-api-key: API-KEY

    x-api-version: 1.0

    You must replace API-KEY with your personal API key.

    This endpoint creates a new invoice.

    HTTP Request

    POST https://www.rapidcoin.com/api/invoices

    JSON Parameters

    Parameter Type Description
    reference_id string, optional Custom reference id associated with invoice
    customer_id string, optional Custom customer id associated with invoice
    description string, optional Custom description associated with invoice
    confirmations_required number, optional Number of blockchains required to complete invoice transaction & trigger confirmation_url IPN
    notification_url string, optional Notification URL to POST when transaction is received on address
    confirmation_url string, optional Notification URL to POST when blockchain confirmations reaches confirmations_required
    redirect_url string, optional URL to redirect to after successful payment (web flow)
    currency (string, optional * - default is BTC) Three character currency code, either USD or BTC. Used with amount to calculate rate & amount_btc
    amount optional * Amount of invoice in specified currency
    * currency and amount are optional, if they are not specified the invoice will be created an "open" invoice

    Response Fields

    Parameter Description
    id Unique Id of invoice
    status Invoice status can either be open, pending, confirming, complete, expired
    token Unique token of invoice
    invoice_time UNIX timestamp when invoice was created
    expiration_time UNIX timestamp of when invoice address is expired of invoice
    exception_status Exception status of invoice
    currency Three character currency code, either USD or BTC. Used with amount to calculate rate & amount_btc
    rate Rate used to convert amount in currency to amount_btc in BTC
    amount Amount of invoice in specified currency
    amount_btc Amount of invoice in BTC, calculated from currency, amount, and rate
    paid_currency Three character currency code, either USD or BTC. Used to show what currency customer will be paid out in
    paid_rate Rate used to convert amount in BTC to paid amount in paid_currency
    paid_amount Amount paid out to customer in paid_currency
    paid_amount_btc Amount paid out to customer in BTC
    redirect_url URL to redirect to after successful payment
    notification_url Notification URL to POST when transaction is received on address
    confirmation_url Notification URL to POST when blockchain confirmations reaches required_confirmations
    confirmations_required Number of blockchain confirmations required to complete invoice and trigger POST to confirmation_notification_url
    reference_id Custom reference id associated with invoice provided on creation
    customer_id Custom customer id associated with invoice provided on creation
    description Custom description associated with invoice provided on creation
    parent_invoice_id Unique Id of parent invoice (if second deposit made on the same address - will reference the first invoice created. null if first invoice)
    paid_btc btc amount paid to invoice
    address Bitcoin deposit address of invoice
    transactions array of blockchain transactions of payments to invoice invoice eg. "transactions": [ {"hash":"f7b0989105b03d69e1ef913b3f7a12df3886d334d0d8706ff77ea78e6602493f", "amount": 0, "confirmations" : 6, "time": 0, "received_time": 0}]
    current_time Current timestamp of request

    Altcoin invoices have similar request parameters to existing invoices, and only modification to the currency parameter is required. To ensure backwards compatibility, the currency codes USD and BTC pass through the original code, which means that handling USD amounts for foreign currencies are not supported. Note that the exchange service may experience downtime for maintenance, and can be checked using the /rates/:currency route (as well as the exchange rate from currency > BTC, and currency > USD). In addition, the confirmations_required parameter is not used, and is set by the altcoin provider. The field is left there as legacy.

    reference_id string, optional Custom reference id associated with payout
    customer_id string, optional Custom customer id associated with payout
    description string, optional Custom description associated with payout

    Response Fields

    Parameter Description
    id Unique Id of payout
    status Payout status can either be pending, complete, exception
    address Address of payout
    currency Three character currency code
    amount Amount of payout to send in specified currency
    amount_subunit Amount of payout to send in the lowest currency precision
    rate_btc Rate used to convert amount in currency to BTC
    rate_usd Rate used to convert amount in currency to USD
    reference_id Custom reference id associated with payout provided on creation
    customer_id Custom customer id associated with payout provided on creation
    description Custom description associated with payout provided on creation
    created_at Datetime payout transaction was created
    completed_at Datetime payout transaction was completed
    transactions Details of blockchain transaction of payout eg. "transaction": {"hash": "f7b0989105b03d69e1ef913b3f7a12df3886d334d0d8706ff77ea78e6602493f"}
